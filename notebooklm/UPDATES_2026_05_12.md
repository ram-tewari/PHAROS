> **Critical updates**: pharos-edge worker supervisor CLI, silent-crash fix in embedding model loader, Windows console-detachment fix, and the first successful end-to-end ingestion in ~25 days. Includes measured benchmark numbers and a Render Starter memory-limit finding.

## Executive Summary

Three things shipped in this update, in dependency order:

1. **Silent-crash root cause identified and patched.** The edge worker had been silently exiting since ~2026-04-27 immediately after `HEAD .../pytorch_model.bin → 302`, with no traceback. Cause: `SentenceTransformer(...)` loading `nomic-embed-text-v1` via the `pytorch_model.bin` path on Windows + Python 3.13 produced a native crash (exit code 3221225477 = ACCESS_VIOLATION) inside nomic's custom `modeling_hf_nomic_bert.py`. The previous handler used a bare `pass` after `logger.error(...)`, so the stack was never persisted.

2. **`pharos-edge` CLI shipped.** Single-file Python tool (`backend/pharos_edge.py`) that replaces the manual `restart_worker.ps1` / `check_worker_status.ps1` dance with `pharos-edge {start | stop | restart | status | logs | run | doctor}`. Includes a subprocess supervisor with exponential backoff, Windows console isolation, and rich-rendered live status.

3. **First clean ingestion in ~25 days.** With the worker actually running, `github.com/spf13/pflag` ingested cleanly: 74 resources, 301 chunks, 0 failed, 86.8s wall-clock. Post-ingest dense retrieval verified working (returns 10 ranked results for parent-child queries). Pre-ingest baseline had returned 0.

## 1. The Silent Crash (Root Cause + Fix)

### What was happening

Every boot of `worker.py` since at least 2026-04-27 stopped logging at this line:

```
2026-05-12 12:17:15,638 - httpx - INFO - HEAD .../pytorch_model.bin "HTTP/1.1 302 Found"
```

The Python process then disappeared. No stderr, no `Traceback`, no `KeyboardInterrupt`. The HuggingFace cache had `pytorch_model.bin` (1 GB) fully populated, so it was not a download issue. The supervisor running for diagnostics observed both flavors of exit:

- exit code **1** (one boot)
- exit code **3221225477** = `0xC0000005` = **ACCESS_VIOLATION** (other boots)

`forrtl: error (200): program aborting due to window-CLOSE event` also appeared in some traces — Intel MKL Fortran runtime intercepting CTRL_CLOSE because the worker subprocess was bound to the parent terminal's console.

### The fix — `backend/app/shared/embeddings.py:69-115`

Three changes:

1. **`model_kwargs={"use_safetensors": True}`** in `SentenceTransformer(...)` so the model loader uses `model.safetensors` instead of `pytorch_model.bin`. Avoids the path that segfaulted.
2. **`faulthandler.enable()`** before the load so any future native crash dumps a stack to stderr instead of vanishing.
3. **`logger.exception(...)`** instead of `logger.error(...)` so the traceback reaches the log. The previous handler swallowed the stack and `pass`ed.

Plus a `TypeError` fallback path for older `sentence-transformers <3` that does not accept `model_kwargs`.

### Companion fix — `backend/worker.py`

Top-of-file additions:
- `faulthandler.enable()` at module load (before any app import)
- `sys.stdout.reconfigure(line_buffering=True)` so log lines surface in real time when stdout is a pipe

### Outcome

After the patch:
- Boot 1: exit 1 after 37s
- Boot 2: exit `3221225477` after 30s (native segfault — caught and respawned)
- **Boot 3: ran stable for 12 minutes with heartbeats every 60s**, both Render (`pharos-cloud-api.onrender.com`) and Upstash returning 200.
- Auto-restart with exponential backoff handled the two early crashes cleanly.

So the safetensors path is not always immediately stable on this hardware either (one early ACCESS_VIOLATION), but the supervisor's backoff makes the worker recover. After three attempts the worker has stayed up.

## 2. The `pharos-edge` CLI

### Why

The prior operator surface for the edge worker was:
- `restart_worker.ps1` (stop existing, start new, foreground)
- `check_worker_status.ps1` (curl + format)
- Manual `python worker.py` for foreground / debugging
- No process supervision; if the worker crashed (it had been crashing silently for ~25 days), nothing restarted it.

That setup is what produced the "every recent ingestion job is `status:"pending"` for 13–25 days" graveyard in `pharos:history`.

### What

Single file: **`backend/pharos_edge.py`**. Subcommands:

| Command | Behavior |
|---|---|
| `pharos-edge start [--wsl]` | Spawn a detached supervisor that owns the worker subprocess. Returns immediately. |
| `pharos-edge stop` | Read `supervisor.pid`, send `SIGTERM` / `CTRL_BREAK_EVENT`, wait 20 s, hard-kill on timeout. |
| `pharos-edge restart [--wsl]` | `stop` then `start`. |
| `pharos-edge status` | Live rich table: supervisor + worker PIDs (with WinAPI liveness probe), mode, restart count, last exit code, cloud heartbeat (auth'd `/api/v1/ingestion/health/worker`), Upstash queue lengths, recent log lines. |
| `pharos-edge logs [-n N]` | Print last N lines of `worker.log`. |
| `pharos-edge run [--wsl]` | Foreground supervisor (used internally by `start`, also useful for systemd / debugging). |
| `pharos-edge doctor` | Validates env vars, Python deps (torch, sentence_transformers, safetensors, rich, dotenv), CUDA, cloud + Upstash reachability, WSL presence. |

### State and logs

All state is kept under `backend/.pharos_edge/` (gitignored):

```
.pharos_edge/
├── supervisor.pid       # int
├── worker.pid           # int
├── state.json           # last snapshot, written every poll
└── logs/
    ├── worker.log       # worker stdout+stderr, 10 MB rotation, keep last 5
    ├── supervisor.log   # supervisor lifecycle (spawns, exits, backoff)
    └── supervisor.out   # detached-process catch-all
```

### Supervisor loop

```
spawn worker
  ↓
poll proc.poll() every 1 s
  ↓ (exit detected)
log exit code, increment restart_count
  ↓
sleep backoff[restart_count]   (5 s → 15 s → 60 s → 300 s → 300 s)
  ↓
respawn

If worker stable >60 s: reset backoff to 0.
```

`SIGINT` / `SIGTERM` / Windows `SIGBREAK` cleanly terminate the worker and write `status: "stopped"` to `state.json` before exiting.

### Windows-specific isolation

The first cut of the supervisor still let `CTRL_CLOSE` on the parent terminal kill the worker (because Intel MKL's `forrtl` traps that signal). Final flags:

| Spawn site | Flags |
|---|---|
| Detached supervisor | `DETACHED_PROCESS \| CREATE_NEW_PROCESS_GROUP \| CREATE_NO_WINDOW \| CREATE_BREAKAWAY_FROM_JOB` |
| Worker subprocess | `CREATE_NEW_PROCESS_GROUP \| CREATE_NO_WINDOW \| CREATE_BREAKAWAY_FROM_JOB` |

Both spawns catch `OSError` and retry without `CREATE_BREAKAWAY_FROM_JOB` (some parent job objects forbid breakaway).

`_pid_alive()` on Windows now uses `OpenProcess(PROCESS_QUERY_LIMITED_INFORMATION) + GetExitCodeProcess` via ctypes — `os.kill(pid, 0)` returned `OSError` for live detached processes, which made `status` falsely report "stopped".

### WSL fallback

`pharos-edge start --wsl` runs the worker via `wsl bash -lc "cd <wsl-path> && python3 -u -X faulthandler worker.py"`. Useful if the Windows-native path becomes unreliable again. Override the WSL Python with `PHAROS_WSL_PYTHON`.

### Dependencies added

`backend/requirements-edge.txt`:
- `rich>=13.0`
- `safetensors>=0.4.0`

No `psutil` dependency — process liveness is checked via stdlib only.

## 3. Benchmarks (real numbers from this session)

All measurements taken from India → `pharos-cloud-api.onrender.com` (Render Starter, us-west / Oregon). Database is Neon `us-east-1` (Virginia) — note the cross-region cloud hop. RTX 4070 *Laptop* GPU (8.0 GB VRAM) hosts the embedding model.

| `POST /api/search/advanced` `parent-child` q="color" | 1.46 s | 1378 ms | **0** |
| `POST /api/search/advanced` `graphrag` q="color" | 0.57 s | 250 ms | 0 (graph empty) |
| `POST /api/search/advanced` `hybrid` q="color" | 1.74 s | 1422 ms | 0 |

`method_contributions: {fts5:6, dense:0, sparse:0}` — every advanced strategy returned 0 because no chunks had embeddings. The retrieval pipeline was effectively dead.

### 3.4 Real bug exposed during ingest — `tree_sitter_go` missing

Worker log spammed this for every `.go` file:

```
WARNING - Failed to load go grammar: Grammar 'go' requires package 'tree_sitter_go'.
          Install with: pip install tree_sitter_go
```

Consequence: **all 301 chunks have `ast_node_type='block'`**, not `function`/`method`/`struct`. The Phase 2 polyglot AST is silently falling back to line-based block chunking for Go because the parser package was never installed. Real chunks in the DB look like:

```
ast_node_type='block'  symbol_name='bool.go:1-50'   semantic_summary='package pflag'
ast_node_type='block'  symbol_name='errors.go:91-140' semantic_summary='func (e *ValueRequiredError) GetSpecifiedShortnames() string {'
```

That is line-chunking with file:line as a synthetic symbol_name. Same gap presumably affects Rust / C / C++ / TS / JS unless their tree-sitter packages are installed.

To verify Phase 2 actually works as designed for any non-Python language, the matching `tree_sitter_<lang>` packages need to be in `requirements-edge.txt`. Currently only Python AST works at function granularity.

### 3.5 Post-ingest search latency

Same `POST /api/search/advanced parent-child` endpoint, now with embeddings populated:

| Query | Wall p50 | Server `latency_ms` | Results | Top sources |
|---|---|---|---|---|
| "boolean flag command line" | 2.70 s | 1874 ms | **10** | langchain (Python `boolean.py`), linux |
| "parse flag default value" | 2.52 s | 2202 ms | 10 | linux |
| "VarP function" | 2.31 s | 2027 ms | 10 | fastapi, linux |
| "ipnet slice" | 2.42 s | 2078 ms | 10 | linux |

Dense retrieval is **actually firing** (results vary by query, sources differ across queries — would all be identical or empty if it weren't). Latency is roughly +650 ms over the dead-pipeline baseline because the HNSW scan now does real work and the response carries 22–32 KB.

### 3.6 Quality finding — pflag chunks don't surface for pflag-relevant queries

None of the four pflag-relevant queries returned a chunk from `github.com/spf13/pflag` in the top 10. The results are dominated by **linux** and **langchain** because:

1. **Block-chunking destroys precision.** A pflag chunk's `semantic_summary` is "package pflag" or a single-line code excerpt — much weaker semantic content than a Python function-level chunk that has signature + docstring + dependency context.
2. **17,714-row corpus drowns 301 new chunks** in dense similarity. Without query-aware reranking or per-source weighting, freshly-ingested narrow content gets buried.
3. Linux's `arch/<arch>/mm/fault.c` etc. simply have richer textual content.

This is a real algorithmic finding, not a benchmark error. Fixing it requires either function-granularity chunking for Go (install `tree_sitter_go`) or a source-prior in the dense ranker.

### 3.7 Strategy comparison — Render Starter hit its memory ceiling

After the four-query parent-child batch and the three-way-hybrid call, repeating the comparison for `q="BoolVar"` returned:

```
GET  /api/search/search/three-way-hybrid q=BoolVar  → HTTP 502 after 11.98 s
POST /api/search/advanced parent-child q=BoolVar    → HTTP 502 / 503
POST /api/search/advanced graphrag q=BoolVar        → HTTP 503
POST /api/search/advanced hybrid q=BoolVar          → HTTP 503
GET  /health                                         → HTTP 503
```

Render dashboard explicitly reported that **`pharos-cloud-api` exceeded its memory limit**. The Starter tier (512 MB) cannot sustain:

- A burst of 12+ search calls each serializing 22–32 KB result bodies (~300 KB total)
- A three-way hybrid that loads 17 k resources for FTS ranking
- An ingestion request that was still cleaning up

Local edge worker survived fine — `pharos-edge status` shows supervisor + worker both running, queues clean, no restarts during the entire benchmark. The bottleneck is the cloud dyno, not the edge.

**Recommendation**: Either move to Render Standard (2 GB), or cap `/api/search/advanced` response bodies (page through chunks instead of returning all surrounding chunks inlined). The current parent-child response carries surrounding chunks + parent resource + scores for every hit, which inflates significantly when `include_code=true`.

## 6. Operational runbook additions

### Start the edge worker

```powershell
# From repo root
.\pharos-edge.ps1 start          # native Windows
.\pharos-edge.ps1 start --wsl    # under WSL (fallback if native crashes)
```

### See what's happening

```powershell
.\pharos-edge.ps1 status         # rich table — refresh as often as you want
.\pharos-edge.ps1 logs -n 100    # last 100 lines of worker.log
.\pharos-edge.ps1 doctor         # env, deps, GPU, cloud + Upstash reachability
```

### Stop / restart

```powershell
.\pharos-edge.ps1 stop
.\pharos-edge.ps1 restart        # stop + start
```

### When something is wrong

1. `pharos-edge status` — what does the supervisor + cloud heartbeat say?
2. `pharos-edge logs -n 200` — any `Traceback` / `Failed to load embedding model` / `exited code=` lines?
3. If the worker is crash-looping on model load, try `pharos-edge restart --wsl`. WSL has historically been more reliable for the embed worker on this machine.
4. If the cloud API is 503-ing on `/health`, the **Render dyno is OOM** — not a worker problem. Wait for Render to auto-restart, then re-run.
